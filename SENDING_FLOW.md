# دليل دورة الإرسال الكاملة

> هذا الملف يصف السلوك الفعلي الموجود في الكود بتاريخ 20 يوليو 2026. لا يفترض سلوكًا غير مطبق، ولا يعني أن أي نمط إرسال يضمن تجنب قيود أو حظر WhatsApp.

## 1. خريطة النظام

يوجد في النظام أربعة مسارات يمكنها إرسال رسالة نصية:

1. **Campaign Queue**: رسائل الحملات المضافة من لوحة التحكم.
2. **AI Cross-Talk**: محادثة ذهاب ورد بين رقمين داخليين مسجلين.
3. **Customer Auto Reply**: رسالة استلام واحدة لعميل أرسل رسالة جديدة أثناء تشغيل النظام.
4. **Direct API Send**: إرسال مباشر بطلب API، خارج الطابور والتأخيرات.

كل إرسال نصي نهائي يمر إلى WAHA عبر:

```text
POST /api/sendText
session = اسم جلسة WAHA
chatId  = الرقم@c.us
text    = نص الرسالة
```

قبل الإرسال، `sendWahaText()` يتحقق من:

- أن اسم الـ Session بالتنسيق المسموح.
- أن الـ Session مستخدمة مع Proxy؛ لا يسمح الإرسال من Session بلا Proxy.

## 2. معنى Start System وStop System

### Start System

يشغّل من الواجهة عمليتين:

- `Campaign Auto-Pilot` لمعالجة طابور الحملات والردود التلقائية.
- `AI Cross-Talk Auto-Pilot` لتشغيل محادثات الأزواج.

ويُسجل الحدث:

```text
SYSTEM / UNIFIED_AUTOPILOT_STARTED
```

إضافة Campaign جديدة من الواجهة تشغّل النظامين تلقائيًا وتسجل نفس الحدث.

### Stop System

- يوقف حلقة Campaign في المتصفح.
- يوقف جدولة محادثات AI الجديدة في المتصفح.
- يسجل `UNIFIED_AUTOPILOT_STOPPED`.
- يحول أي `AUTO_REPLY` ما زال `PENDING` إلى `CANCELLED`، حتى لا يُرسل عند دورة تشغيل لاحقة.

حالة التشغيل تُسجل في قاعدة البيانات. Campaign وCustomer Auto Reply وAI Cross-Talk تعالجها Backend Schedulers حتى لو أُغلقت صفحة Control Room؛ المطلوب فقط بقاء Backend نفسه يعمل.

## 3. الشروط العامة لاختيار رقم مُرسل

الرقم لا يدخل في الإرسال الآلي إلا إذا اجتمعت الشروط التالية:

- موجود في قاعدة `WahaSender`.
- حالته `ACTIVE`.
- جلسة WAHA الخاصة به حالتها `WORKING`.
- رقم الحساب الحقيقي الذي تعيده WAHA يساوي الرقم المسجل في قاعدة البيانات.
- الرقم ليس مشغولًا حاليًا بعملية أخرى داخل نفس عملية Node.
- في الحملات: لم يصل إلى `maxDailyLimit`.

تزامن الـ Monitor يجعل الجلسات غير العاملة `OFFLINE`. عند عودة Session إلى `WORKING` يرجع السجل `ACTIVE` فقط إذا كانت حالته السابقة `OFFLINE`؛ ولا يلغي `BANNED` أو `RESTING` يدويًا.

يوجد Mutex في الذاكرة:

- Campaign أو Auto Reply يقفل المرسل حتى نهاية Job.
- Cross-Talk يقفل الرقمين حتى نهاية المحادثة.
- الهدف منع استخدام نفس الرقم في عمليتين متزامنتين داخل نفس عملية السيرفر.

هذا القفل غير موزع بين أكثر من Backend، ويُمسح عند إعادة تشغيل Node.

## 4. Campaign Queue — المسار الأساسي للحملات

### 4.1 إضافة الحملة

من الواجهة يتم إرسال:

```text
POST /api/campaign/queue
```

ما يحدث:

1. قراءة اسم الحملة، الأرقام، وقالب الرسالة.
2. تقسيم الأرقام باستخدام السطر الجديد أو المسافة أو `,` أو `;`.
3. إزالة التكرارات داخل نفس الطلب.
4. إنشاء Job لكل Target بحالة `PENDING` ونوع `CAMPAIGN`.
5. تسجيل `CAMPAIGN_QUEUED` في Activity Log.
6. تشغيل Campaign وAI Auto-Pilot تلقائيًا من الواجهة.

قالب الرسالة يدعم Spintax بسيطًا:

```text
{أهلاً|مرحبًا} بحضرتك
```

كل مجموعة `{...}` تختار اختيارًا واحدًا عشوائيًا وقت معالجة الـ Job.

### 4.2 توقيت استدعاء الـ Worker

الواجهة تستدعي:

```text
POST /api/campaign/worker
```

- إذا لم يوجد Job جاهز: يعيد المحاولة بعد **5 ثوانٍ**.
- بعد معالجة Job: ينتظر مدة عشوائية بين `Campaign minimum delay` و`Campaign maximum delay`.
- القيم الافتراضية في Backend: **30,000–90,000 ms**، أي **30–90 ثانية**.
- يمكن تغييرها عبر `CAMPAIGN_MIN_DELAY_MS` و`CAMPAIGN_MAX_DELAY_MS` ثم إعادة تشغيل Backend.

هذه المدة بين استدعاءات الـ Worker، وتُضاف إلى وقت `seen/typing/send` داخل الـ Job.

### 4.3 اختيار Job

الـ Worker يبحث عن:

- أقدم `AUTO_REPLY` وصل إلى `scheduledAt`.
- وأقدم `CAMPAIGN` جاهز.

إذا كان النوعان جاهزين، يتناوب بينهما:

```text
AUTO_REPLY → CAMPAIGN → AUTO_REPLY → CAMPAIGN
```

لا توجد قاعدة «بعد كل 5 رسائل». التناوب يحدث فقط عندما يكون النوعان جاهزين في اللحظة نفسها. إذا لم يوجد Auto Reply جاهز، تستمر الحملات عاديًا.

### 4.4 اختيار Sender للحملة

يُستخدم Round-Robin بين الأرقام `ACTIVE + WORKING` وغير المشغولة وتحت الحد اليومي. الترتيب الأساسي حسب رقم الهاتف، ومؤشر Round-Robin يبدأ بقيمة عشوائية عند تشغيل السيرفر.

### 4.5 خطوات إرسال Campaign Job

```text
PENDING
  ↓ claim
PROCESSING
  ↓ اختيار Sender
فحص Proxy
  ↓
Seen
  ↓ 0.5–1.499 ثانية
Typing
  ↓ 2–20 ثانية حسب النص
Paused
  ↓ 0.2–0.699 ثانية
SendText
  ↓
DONE أو FAILED
```

التفصيل:

1. تغيير Job من `PENDING` إلى `PROCESSING`.
2. تسجيل `JOB_CLAIMED`.
3. اختيار Sender صالح.
4. إذا لم يوجد Sender: إعادة Job إلى `PENDING` وتسجيل `JOB_REQUEUED`.
5. قفل Sender.
6. فحص TCP للـ Proxy بحد أقصى **8 ثوانٍ**.
7. استدعاء `sendSeen`.
8. انتظار عشوائي **500–1499 ms**.
9. ضبط Presence إلى `typing`.
10. انتظار وقت الكتابة المحسوب.
11. ضبط Presence إلى `paused`.
12. انتظار عشوائي **200–699 ms**.
13. إرسال الرسالة عبر WAHA.
14. عند النجاح: Message Log = `SENT`، Job = `DONE`، وتسجيل `MESSAGE_SENT`.
15. عند الفشل: Message Log = `FAILED`، Job = `FAILED` مع السبب، وتسجيل `MESSAGE_FAILED`.
16. فك قفل Sender في كل الحالات.

### 4.6 وقت الكتابة المحسوب

الدالة `calculateTypingTime(text)` تعمل كالتالي:

```text
عدد الحروف × قيمة عشوائية من 80 إلى 250 ms
+ اختلاف عشوائي ±25%
+ وقت استجابة من 800 إلى 2499 ms
ثم حصر الناتج بين 2 و20 ثانية
```

إذًا زمن Campaign Job الداخلي قبل الإرسال، بدون Proxy أو WAHA network، تقريبًا:

```text
Seen gap 0.5–1.499s
+ typing 2–20s
+ paused gap 0.2–0.699s
= 2.7–22.198s
```

ثم يُضاف فحص Proxy حتى 8 ثوانٍ، وزمن اتصال WAHA، وبعد Job تُضاف مدة الواجهة 30–90 ثانية افتراضيًا.

### 4.7 الحالات والفشل

- `PENDING`: ينتظر.
- `PROCESSING`: الـ Worker استلمه.
- `DONE`: WAHA أكد نجاح طلب الإرسال.
- `FAILED`: فشل قبل أو أثناء الإرسال.
- `STUCK`: عرض فقط عندما يظل `PROCESSING` أكثر من دقيقتين.
- `CANCELLED`: مستخدمة للـ Auto Reply الملغي عند Stop System.

`Retry All Failed` يعيد Jobs الفاشلة إلى `PENDING`. هذا قد يكرر رسالة إذا كانت وصلت إلى WhatsApp لكن الاستجابة ضاعت قبل تسجيل `DONE`.

`Recover Stuck` يعيد `PROCESSING` الأقدم من دقيقتين إلى `PENDING`، وله نفس احتمال التكرار.

## 5. Customer Auto Reply

### 5.1 ما الذي يُنشئ الرد؟

WAHA يرسل Webhook إلى:

```text
POST /api/webhook
event = message أو message.any
```

لا يفحص النظام قائمة المحادثات، ولا unread messages، ولا Chat History للعملاء. القرار مبني فقط على Webhook وصل الآن.

### 5.2 شروط القبول كاملة

يُنشأ Auto Reply فقط إذا:

- الرسالة ليست `fromMe`.
- المرسل Chat فردي وينتهي بـ `@c.us`؛ الجروبات وStatus لا تدخل.
- للرسالة Message ID.
- للحدث Session Name.
- آخر حدث System هو `UNIFIED_AUTOPILOT_STARTED`.
- Timestamp الرسالة يساوي أو يأتي بعد وقت Start System الحالي.
- المرسل ليس واحدًا من أرقام `WahaSender` الداخلية.
- لم يُنشأ لهذا العميل على نفس Session رد `PENDING/PROCESSING/DONE` منذ Start System الحالي.

بالتالي:

- الرسائل القديمة عند فتح WhatsApp: **لا رد**.
- الرسائل التي وصلت والنظام متوقف: **لا رد**.
- عدة رسائل من العميل في دورة التشغيل نفسها: **رد واحد فقط**.
- تكرار `message` و`message.any` لنفس Message ID: **لا تكرار** بسبب `sourceMessageId UNIQUE`.
- رقم داخلي يرسل لرقم داخلي: **لا Auto Reply**؛ هذا يمنع loop مع Cross-Talk.
- تشغيل النظام في يوم لاحق ثم وصول رسالة جديدة من العميل نفسه: يمكن إنشاء رد واحد جديد للدورة الجديدة.

### 5.3 التأخير والرسالة

عند قبول الرسالة:

1. اختيار نص واحد من ثلاثة نصوص استلام ثابتة.
2. اختيار تأخير عشوائي افتراضي **35–120 ثانية**.
3. إنشاء Job نوعه `AUTO_REPLY` واسم حملته `Customer Auto Replies`.
4. تسجيل `AUTO_REPLY / REPLY_QUEUED` وفي Metadata: الـ Session والعميل والتأخير وMessage ID.

يمكن تغيير المدة بمتغيرات البيئة:

```env
AUTO_REPLY_MIN_DELAY_SECONDS=35
AUTO_REPLY_MAX_DELAY_SECONDS=120
```

### 5.4 كيف يُرسل؟

بعد وصول `scheduledAt` يعالجه نفس Campaign Worker، لكن Sender ليس Round-Robin: يجب أن يكون **نفس الـ Session التي استقبلت رسالة العميل**، وأن تكون `ACTIVE + WORKING`.

بعد ذلك يمر بنفس التسلسل:

```text
Proxy check → Seen → 0.5–1.499s → Typing → 2–20s
→ Paused → 0.2–0.699s → SendText
```

الزمن الفعلي من وصول رسالة العميل ليس 35–120 ثانية فقط. قد يُضاف:

- حتى 5 ثوانٍ Polling عندما يكون الطابور فارغًا.
- انتظار الـ Worker الحالي إن كان مشغولًا.
- مدة 30–90 ثانية بين دورات Worker حسب إعداد الواجهة.
- 2.7–22.198 ثانية لإشارات الشات والكتابة.
- حتى 8 ثوانٍ لفحص Proxy، إضافة إلى زمن WAHA والشبكة.

### 5.5 Stop System

عند Stop System تتحول Auto Replies المنتظرة من `PENDING` إلى `CANCELLED`. لا تُرسل في Start قادم. الرسالة التي أصبحت `PROCESSING` بالفعل ليست ضمن عملية الإلغاء الحالية، لذلك الضغط على Stop أثناء تنفيذها لا يعد إلغاءً ذريًا للإرسال الجاري.

## 6. AI Cross-Talk

### 6.1 متطلبات التشغيل

- وجود رقمين على الأقل `ACTIVE + WORKING`.
- تطابق Session مع رقم الحساب الحقيقي في WAHA.
- الرقمان غير مشغولين بعملية أخرى.
- Proxy الرقمين ينجح في TCP check؛ كل فحص يمكن أن يستغرق حتى 8 ثوانٍ.

إذا كان المتاح أقل من رقمين يسجل `WAITING_FOR_SENDERS` ولا يرسل.

### 6.2 اختيار الأزواج

لعدد `N` من الأرقام، النظام يبني كل الأزواج غير المرتبة وعددها:

```text
N × (N - 1) / 2
```

مثال 4 أرقام = 6 أزواج:

```text
A↔B, A↔C, A↔D, B↔C, B↔D, C↔D
```

يتم اختيار الزوج الأقل تواصلًا مؤخرًا. الزوج الذي لم يتواصل من قبل له أولوية، لذلك تغطى كل الأزواج قبل تكرار الأزواج القديمة. داخل تشغيل واحد للزوج، الاتجاه هو Sender 1 ثم Sender 2 يرد؛ في مجموع التشغيلات لا توجد حاليًا ذاكرة منفصلة تضمن عكس مَن يبدأ كل مرة.

### 6.3 إنشاء النص بالـ AI

المزود الحالي هو Groq والنموذج:

```text
llama-3.3-70b-versatile
temperature = 0.9
```

الرسالة:

- من 3 إلى 10 كلمات وفق الـ prompt.
- من 1 إلى 3 Bubbles باستخدام `|||`.
- يستخدم آخر 6 رسائل ناجحة بين الزوج كـ Chat History عند بداية الحديث.
- احتمال 20% لتغيير الموضوع عند الرد.
- احتمال 10% لطلب typo ثم Bubble تصحيح منفصلة.

هذه قواعد Prompt؛ استجابة النموذج قد لا تلتزم بها دائمًا حرفيًا.

### 6.4 التسلسل الكامل

```text
اختيار الزوج
→ فحص Proxy للرقمين
→ قفل الرقمين
→ قراءة آخر 6 رسائل
→ توليد رسائل S1
→ S1 Seen
→ انتظار 5–20s
→ [Typing → 2–20s → Paused → 2–6s → Send] لكل Bubble
→ بين Bubbles: 3–12s
→ انتظار رد: غالبًا 15–60s، وأحيانًا 60–180s
→ S2 Seen
→ قراءة 5–20s
→ توليد رد S2
→ [Typing → 2–20s → Paused → 2–6s → Send] لكل Bubble
→ بين Bubbles: 3–12s
→ انتظار 10–30s
→ S1 Seen
→ تسجيل CONVERSATION_COMPLETED
→ فك قفل الرقمين
```

نافذة انتظار رد S2:

- احتمال 80%: **15–60 ثانية**.
- احتمال 20%: **60–180 ثانية**.

بعد اكتمال محادثة زوج، Auto-Pilot ينتظر قبل اختيار محادثة أخرى:

- افتراضيًا **30–120 ثانية**.
- عند Groq `429` يستخدم `retry-after` أو المدة المستخرجة من الخطأ، مع زيادة ثانيتين؛ والافتراضي 15 ثانية إذا لم توجد مدة.

### 6.5 زمن محادثة AI التقريبي

باستبعاد زمن Groq، فحص الـ Proxy، WAHA، والشبكة:

- أقصر سيناريو نظري، Bubble واحدة لكل طرف ونافذة الرد القصيرة: حوالي **43 ثانية**.
- أطول سيناريو داخل الحدود الحالية، 3 Bubbles لكل طرف ونافذة distracted القصوى: حوالي **7 دقائق و34 ثانية**.

فحص Proxy للرقمين متتالٍ وقد يضيف حتى **16 ثانية**، كما يضاف زمن طلبي Groq وزمن WAHA.

كل Bubble ناجحة تسجل في `MessageLog` وتزيد `dailySentCount` للرقم الذي أرسلها. Bubble الفاشلة تسجل `FAILED` بتفاصيل WAHA وتطبق نفس تصنيف `BANNED/RESTING` المستخدم في Campaign، ثم تتوقف المحادثة ويُفك قفل الرقمين.

## 7. Seen وRead وTyping وPaused

### 7.1 الوضع الحالي الفعلي

هذه الإشارات اختيارية ومفعلة حاليًا بإعداد التشغيل التالي:

```env
WAHA_ENABLE_CHAT_SIGNALS=true
```

إذا تم تغيير القيمة أو حذفها تتوقف إشارات الشات، بينما تظل فترات الانتظار والإرسال تعمل.

### 7.2 معنى كل إشارة

- `sendSeen`: يطلب من WAHA تعليم الشات كمقروء وإرسال read receipt حسب إعدادات WhatsApp والطرفين.
- `typing`: يطلب إظهار أن الحساب يكتب داخل الشات.
- `paused`: يوقف حالة الكتابة.

هذه الإشارات ليست شرطًا لنجاح الرسالة. لو WAHA WEBJS أعاد 500 أو 404 تسجل Warning في Console ولا تُفشل عملية الإرسال. السبب أن Presence قد يفشل عندما لا يكون الشات موجودًا محليًا بعد.

## 8. Direct API Send

المسار:

```text
POST /api/waha/send
{
  "sessionName": "...",
  "phoneNumber": "...",
  "message": "..."
}
```

هذا المسار:

- يتحقق من وجود الحقول.
- يتحقق داخل `sendWahaText` من اسم Session ووجود Proxy.
- يرسل مباشرة إلى WAHA.
- لا يدخل Campaign Queue.
- لا يعمل Round-Robin.
- لا يعمل Proxy TCP health check المنفصل.
- لا يضيف تأخيرًا.
- لا يعمل Seen أو Typing أو Paused.
- لا ينشئ `MessageLog` أو `ActivityLog` حاليًا.

لذلك هو API منخفض المستوى للاختبار أو التكامل المباشر، وليس نفس مسار Auto-Pilot.

## 9. مسار Campaign القديم المتزامن

يوجد أيضًا:

```text
POST /api/campaign/start
```

هذا مسار متزامن أقدم ومختلف عن Queue المستخدم في الواجهة:

- حد أقصى 100 Target.
- ينظف الأرقام ويقبل 8–15 رقمًا.
- Round-Robin مع محاولة عدم تكرار Proxy السابق مباشرة.
- يفحص Proxy.
- يرسل `typing` فقط، بدون `seen` وبدون `paused`.
- وقت الكتابة فيه `min(length × 100ms, 4s)`.
- التأخير الافتراضي بين الأهداف 3–8 ثوانٍ، ومحصور من 1–60 ثانية عبر API.
- الطلب يظل مفتوحًا حتى انتهاء Batch.

لو الهدف هو النظام الحالي الموحد، استخدم `/api/campaign/queue` مع `/api/campaign/worker`، وليس `/api/campaign/start`.

## 10. السجلات والعدادات

### MessageLog

يسجل لكل رسالة أرسلها Campaign Worker أو Cross-Talk:

- Sender.
- Target.
- نص الرسالة.
- `SENT` أو `FAILED`.
- سبب الفشل.
- وقت الإنشاء.

عند `SENT`:

- يزيد `dailySentCount` بمقدار 1 لكل Bubble.
- يتحدث `lastActiveAt`.

عند `FAILED`:

- أخطاء واضحة مثل banned/403/401 قد تحول Sender إلى `BANNED`.
- أخطاء Session المغلقة أو المفقودة قد تحوله إلى `RESTING`.

### ActivityLog / Live Activity

المصادر:

- `SYSTEM`
- `CAMPAIGN`
- `CROSS_TALK`
- `AUTO_REPLY`

ويحفظ الحدث والحالة والوصف والـ Metadata مثل Sender وTarget وProxy وMessage Body وJob ID.

الواجهة تجلب Monitor كل **5 ثوانٍ**، وحالة Queue كل **10 ثوانٍ**؛ لذلك ظهور الأحداث والأرقام في الصفحة ليس لحظيًا 100% وقد يتأخر حتى دورة التحديث التالية.

## 11. السيناريوهات المختصرة

| السيناريو | النتيجة |
|---|---|
| إضافة Campaign جديدة | تُضاف Jobs ويبدأ Campaign وAI من الواجهة |
| Campaign + Auto Reply جاهزان | تناوب بين النوعين، بدون قاعدة كل N رسائل |
| العميل يرسل أثناء Start System | رد واحد مؤجل من نفس Session |
| العميل يرسل عدة Bubbles في نفس الدورة | رد واحد فقط |
| فتح Session وبها رسائل قديمة | تجاهل كامل بسبب Timestamp |
| وصول رسالة والنظام متوقف | تجاهل كامل |
| Stop System والرد ما زال Pending | يتحول إلى CANCELLED |
| رسالة من رقم داخلي | لا Auto Reply لتجنب loop |
| أقل من رقمين صالحين | Cross-Talk لا يعمل |
| Proxy يفشل | Job يفشل أو Cross-Talk يُلغى قبل الإرسال |
| Session ليست WORKING | لا تدخل في اختيار Sender |
| Presence/Seen يفشلان | يستمر الإرسال؛ الإشارة تجميلية وليست شرطًا |
| WAHA يؤكد الإرسال | DONE + SENT logs + زيادة العداد |
| استجابة الإرسال تفشل | FAILED مع السبب |
| Groq يعيد 429 | Cross-Talk ينتظر المدة المحددة ثم يحاول لاحقًا |
| استدعاء Direct Send | إرسال فوري خارج كل الطوابير والتأخيرات |

## 12. حدود وضمانات مهمة

- نجاح WAHA `sendText` يعني أن WAHA قبل عملية الإرسال؛ لا يثبت أن المستلم قرأ الرسالة.
- `seen` و`typing` قد لا يعملا مع WEBJS لبعض الشاتات، ولا يمنع ذلك الإرسال.
- لا توجد طريقة تقنية تضمن عدم فرض WhatsApp قيودًا أو حظرًا. الالتزام بموافقة المستلمين وسياسات المنصة ومعدلات إرسال معقولة يظل مسؤولية التشغيل.
- Campaign وAuto Reply وAI Cross-Talk تعتمد على Backend Schedulers دائمة ولا تحتاج لبقاء الصفحة مفتوحة.
- Retry وRecover قد يسببان Duplicate في حالات انقطاع الشبكة بعد وصول الرسالة وقبل حفظ النتيجة.
- تغيير `CAMPAIGN_MIN_DELAY_MS` و`CAMPAIGN_MAX_DELAY_MS` يؤثر في Backend Worker بعد إعادة تشغيله، ولا يغير تأخير AI أو موعد Auto Reply الأولي.
